# termux_flutter Detailed Roadmap

## PHASE 0 — Project Foundation & Repository Setup

This phase exists because a Termux-compatible terminal cannot be built safely as a collection of isolated fixes. The current repository mixes a Flutter application, Android native shell bridge code, and an upstream `termux-app/` reference tree, so the first job is to define which sources are authoritative and which are reference-only. The work should pin Flutter, Dart, Android Gradle Plugin, Gradle, JDK, CMake, and NDK versions, then document device targets such as `android-arm`, `android-arm64`, `x86`, and `x86_64`.

The implementation should add repeatable build commands, source inventory rules, package identity policy, and acceptance gates tied to Termux features rather than visual demos. This phase should also state the risk around Android 5.1/API 22 and API 21 experiments, because modern Flutter engine behavior, NDK output, and Java library desugaring can behave differently on old devices. Dependencies are mostly documentation and tooling, but they unblock every later phase. Without this foundation, PTY, bootstrap, emulator, and lifecycle work can appear to pass on one machine while failing silently on another.

## PHASE 1 — PTY & Native Terminal Core

This phase addresses the root cause behind Android shell fallback. A Termux-like app must not treat `/system/bin/sh` as the normal interactive shell; it must allocate a PTY, fork or spawn a child process, attach the child to the controlling terminal, configure termios, set environment variables, and execute the intended app-private shell. The current project already has native and Java shell bridge pieces, including `pty_bridge.c`, `PtyProcess.java`, `ShellEngine.java`, and channel code, but those pieces need to be validated against upstream Termux `termux.c`.

Implementation should use JNI/NDK for the low-level terminal process path and Flutter `MethodChannel`/`EventChannel` for the UI bridge. Required behavior includes `/dev/ptmx`, `grantpt`, `unlockpt`, `TIOCSCTTY`, UTF-8 termios, binary-safe reads and writes, resize propagation, signals, exit status, and fd cleanup. The largest risk is accepting a pipe-based or Android-shell fallback as "good enough"; that will break full-screen programs, signals, curses apps, and package manager behavior. This phase depends on Phase 0 build reproducibility and prepares the execution substrate for bootstrap and session lifecycle work.

Phase 1 implementation note: `PTY_GAP_ANALYSIS.md` records the comparison against upstream `termux.c`. The local bridge now opens `/dev/ptmx` directly, configures raw UTF-8/no-flow-control termios, applies startup and resize `TIOCSWINSZ` with cell pixel dimensions, starts the child as a session leader with `TIOCSCTTY`, closes inherited fds, streams PTY output as bytes through Java and Flutter, exposes `sendSignal`, and removes pipe fallback from `ShellEngine`. Instrumentation tests live in `android/app/src/androidTest/java/com/termux/flutter/PtyProcessInstrumentationTest.java`. Validation command: `cd android && gradlew.bat :app:connectedAndroidTest`.

## PHASE 2 — VT100/xterm Terminal Emulator (Dart layer)

This phase replaces the limited ANSI parsing approach with a real terminal emulator. A Termux-compatible app has to understand VT100/xterm behavior, not just colored text. Programs such as `vim`, `nano`, `less`, `top`, `htop`, `apt`, shells, readline, and curses applications rely on cursor movement, alternate screen, scroll regions, OSC, DCS, bracketed paste, character sets, Unicode width, combining marks, and many SGR attributes.

The project should evaluate the Flutter `xterm` package against upstream `terminal-emulator` behavior before deciding whether to adopt it or port Termux's `TerminalEmulator.java` concepts into Dart. The emulator should expose a renderer-neutral screen model so the Flutter terminal view does not parse escape sequences directly. Tests should be migrated from upstream wherever practical, especially around color, cursor, scrollback, Unicode width, and screen mutation. Risks include subtle incompatibilities that only appear under real terminal workloads and performance issues when high-volume output arrives through the PTY. This phase depends on Phase 1 for reliable input/output streams and feeds Phase 3 rendering.

Phase 2 implementation note: `decision_emulator.md` records the `xterm 4.0.0` evaluation and the decision to port Termux emulator concepts into project-owned Dart types. The core implementation lives in `lib/core/terminal/` and exposes `TerminalEmulator`, `ScreenModel`, `ScreenCell`, `TextAttributes`, `ColorAttribute`, `WcWidth`, `TerminalKeys`, and `AlternateScreenManager`. The live Flutter controller now uses `lib/terminal/terminal_emulator_adapter.dart` to translate the renderer-neutral cells into the existing painter's `TerminalBuffer`, leaving PTY, bootstrap, service, and rendering phases untouched. Tests under `test/terminal/` cover upstream-derived CSI, OSC, DCS/APC, alternate screen, scroll region, SGR color, Unicode width, and screen model behavior. Root cause was the old `AnsiParser` mixing a tiny escape subset with Flutter `TextStyle` storage. Source references: upstream `TerminalEmulator.java`, `TerminalBuffer.java`, `WcWidth.java`, and `terminal-emulator` tests. Validation command: `flutter test test/terminal/ --reporter=expanded`.

## PHASE 3 — Terminal View & Rendering Engine

This phase turns emulator state into a usable Flutter terminal interface. The current Flutter surface uses a custom painter and a small parser, which is not enough for Termux parity. The target view should render emulator rows, transcript rows, attributes, cursor state, scrollback, and selections with stable glyph metrics. It should handle ASCII, CJK, emoji, combining characters, box drawing, bold, faint, italic, underline, inverse, strike, foreground colors, background colors, and cursor styles.

The work should be implemented in Flutter, with careful measurement and caching rather than rebuilding the whole screen for every byte of output. It should also reproduce the expected mobile terminal interactions: fling scrolling, scroll-to-bottom, long-press selection, drag handles, copy, paste, font scaling, and persistent font preferences. The main risks are jank under high output, incorrect cell widths, and text overlap on small screens. This phase depends on Phase 2's screen model and should use upstream Termux `TerminalView.java` and `TerminalRenderer.java` as behavioral references rather than direct UI copies.

## PHASE 4 — Bootstrap Filesystem & Package Manager Bridge

This is the phase that makes `pkg`, `apt`, and `dpkg` possible. The current project has a `BootstrapInstaller` scaffold that expects ABI-specific bootstrap archives, but a scaffold alone is not a Linux userspace. A Termux-like app needs a valid app-private `$PREFIX`, `$HOME`, repository configuration, executable permissions, symlinks, dynamic linker paths, and a shell such as `$PREFIX/bin/bash`. It also has to avoid relying on `/system/bin/sh` as the user's terminal shell.

Implementation should port the important behavior from upstream `TermuxInstaller`: verify the bootstrap archive, extract into a staging directory, read and create `SYMLINKS.txt`, chmod executables, atomically move the completed prefix into place, and write the environment expected by sessions. Environment variables should include `PREFIX`, `HOME`, `TMPDIR`, `SHELL`, `PATH`, `LD_LIBRARY_PATH`, `LANG`, and Android-specific values needed by Termux packages. Risks include ABI mismatches, old Android API incompatibilities, broken symlinks on extraction, and repository metadata that does not match the bootstrap. This phase depends on Phase 1 and directly enables package manager validation.

## PHASE 5 — Shell Session Management & Lifecycle

This phase moves terminal processes out of fragile Activity ownership and toward Termux-style persistent session management. Real Termux keeps terminal sessions alive through `TermuxService`, supports multiple sessions, manages process lifecycle, and exposes foreground notifications when required by Android. A Flutter Activity can render and control a session, but it should not be the sole owner of the shell process if the app rotates, backgrounds, or is recreated.

Implementation should introduce an Android `ForegroundService` or equivalent service-backed manager that owns PTY sessions, session IDs, resize state, title changes, process exit events, and reconnect behavior. Flutter can communicate through `MethodChannel`, `EventChannel`, or a generated Pigeon API if the channel surface grows complex. The service should support create, list, attach, detach, rename, close, and kill operations. Risks include leaks, duplicate sessions after Activity recreation, notification policy failures on modern Android, and confusing behavior on older Android versions. This phase depends on stable PTY startup from Phase 1 and becomes more useful once Phase 4 provides a real shell.

## PHASE 6 — Keyboard System (Extra Keys Row + Hardware Keyboard)

This phase makes the terminal practical on phones. Android soft keyboards are not designed for shells, so Termux provides extra keys, hardware keyboard mapping, modifier handling, paste controls, and terminal-specific escape generation. The current Flutter widget can request text input, but a Termux-like terminal needs reliable Ctrl, Alt, Esc, Tab, arrows, function keys, page keys, and combinations used by shells and terminal applications.

Implementation should study upstream `KeyHandler.java`, `TerminalViewClient`, and Termux extra keys behavior. The Flutter side should expose an extra-keys row and translate taps or hardware key events into the correct byte sequences for the PTY. It should handle IME composition carefully so normal text entry works while terminal control keys remain exact. Risks include Android keyboard variance, old-device IME bugs, focus loss, and accidental use of desktop-oriented Flutter shortcuts that do not map to terminal bytes. This phase depends on Phase 1 write-path correctness and Phase 3 focus/rendering stability, and it improves validation for every interactive command.

## PHASE 7 — Theming, Fonts & Visual Polish

This phase brings the app from functional terminal to daily-driver terminal. Termux users expect readable fonts, stable cell alignment, dark and light themes, color scheme customization, cursor options, and visual behavior that remains consistent across devices. The current project should avoid treating visual polish as decoration; in a terminal, font metrics and color contrast are correctness issues because they determine whether TUIs and prompts are readable.

Implementation should build a theme model in Flutter that maps emulator colors, default foreground/background, cursor color, selection color, font family, font size, line height, and bold rendering. Settings should be persistent and should not break glyph alignment. The renderer must be tested with ASCII, box drawing, Unicode, emoji, CJK, and truecolor output. Risks include choosing a font that lacks glyph coverage, using proportional metrics by accident, and creating layouts that resize unexpectedly when terminal content changes. This phase depends on the Phase 3 renderer and should be validated with the same terminal programs used for emulator testing.

## PHASE 8 — Termux:API Plugin Bridge

This phase addresses integration with Android device APIs. Termux:API exposes commands that let shell scripts interact with Android features through separate app components and command-line tools. A Flutter clone that only runs a shell is incomplete if the common `termux-*` commands cannot communicate with an Android-side implementation or compatible plugin package.

Implementation should define a bridge between the Linux userspace and Android services. Depending on compatibility goals, this may involve supporting upstream Termux:API package expectations, Android intents, binder/service calls, or a Flutter/Android command bridge. The app should document package names, permissions, result formats, and failure behavior. `MethodChannel` may be enough for UI-driven actions, but service-level APIs should live on the Android side so shell commands work even when Flutter UI is not actively rendering. Risks include permission drift across Android versions, plugin package identity conflicts, and security mistakes when exposing device capabilities to shell processes. This phase depends on Phase 4 package installation and Phase 5 service lifecycle.

## PHASE 9 — Termux Plugin Ecosystem (Widget, Float, Boot, Tasker, X11)

This phase expands beyond the main terminal and considers the broader Termux ecosystem. Users often rely on Termux:Widget, Termux:Boot, Termux:Float, Tasker integration, and X11-related workflows. Matching those features exactly is a large compatibility project, so this phase should define what "supported" means for each plugin and where the Flutter app intentionally diverges.

Implementation should begin with package identity and intent compatibility analysis. Widget and Boot features require Android receivers and background execution policy handling. Float requires overlay/window behavior and careful Android permission flow. Tasker requires stable intent contracts. X11 support may depend on external packages, display sockets, and rendering assumptions outside the Flutter terminal itself. The app should prefer compatibility shims where upstream plugin behavior can be preserved and explicit documentation where it cannot. Risks include Android background restrictions, Play policy implications, overlay permission complexity, and maintaining compatibility with upstream plugin updates. This phase depends on stable sessions, a working userspace, and a plugin bridge foundation from earlier phases.

## PHASE 10 — Full Feature Parity Validation & QA

This final phase proves that the app behaves like Termux in practice, not just in isolated tests. Validation should cover first install, bootstrap, shell launch, `pkg update`, `apt install`, PTY behavior, terminal rendering, keyboard input, session persistence, storage, plugins, and old-device compatibility. It should include multiple ABIs and Android versions, especially any claimed Android 5.1/API 22 support.

Implementation should combine automated tests, manual test scripts, golden terminal behavior checks, APK install tests, and real-device smoke tests. The validation matrix should include commands such as `bash --login`, `pkg update`, `apt update`, `apt install`, `dpkg -l`, `vim`, `nano`, `less`, `top`, Ctrl+C, resize, paste, scrollback, Activity recreation, and background/foreground recovery. Risks include passing debug builds but failing release builds, ABI-specific native crashes, missing bootstrap files, and silent fallback to `/system/bin/sh`. This phase depends on every previous phase and should be treated as the release gate for claiming Termux-like behavior.
